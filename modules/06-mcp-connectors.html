MCP & connectors

How an agent reaches the outside world: a standard plug, not bespoke wiring.

This answers "how does it touch my files, calendar or database?" Frame MCP as the USB-C of agent tooling: one protocol instead of N custom integrations.

Why a shared protocol?

If many assistants must connect to many tools, every bespoke connection is extra work. A shared protocol gives everyone one common socket.

Many bespoke connectionsWithout a standard, every app-tool pair needs its own dedicated code.
With MCPAgents and tools speak the same language of request and response.
Not merging dataMCP moves requests, results and capability descriptions; it does not merge your data stores.
Explain aloud: many assistants connecting to many tools. Everything is now plain language: too many bespoke connections are the reason a common standard reduces the work.

Model Context Protocol

An open standard: an application connects to services that expose their capabilities in a uniform way.

MCP standardises the way an agent asks for tools, data and instructions, so it can discover for itself what a service offers.

"open standard for connecting AI assistants"Anthropic announcement
"standardizes how applications provide context"MCP introduction

Connectors in practice

DiscoverThe connector advertises available tools, resources and schemas.
CallThe agent sends a structured request through the protocol.
ReturnThe connector returns results that become new context for the model.
Connectors are where a chatbot becomes a colleague that does things.

Same security questions as any integration: credentials, scope and trust.
